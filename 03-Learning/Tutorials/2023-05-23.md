我的

v1

```
公司为旗下酒店提供管理后台以及面向消费者的网页和App，包括订单、财务、合同、报表分析和IOT等模块。

1、负责将遗留的财务服务迁移到新系统中

- 遗留的财务服务使用Spring Boot+Kotlin开发，已经运行了三年多。该服务仍然勉强可用，但原先的开发人员已经离职，没有留下足够的文档，维护变得非常困难。服务中的一些关键设计也已经不能满足财务业务上的需要。同时，整个后端服务除了该服务，都是采用Scala+Akka开发，同时存在两套技术栈不利于统一维护。基于上述情况，由我统筹负责将遗留的财务服务迁移到新系统中。

- 迁移过来的新服务在兼顾老服务行为与业务的同时，产品经理也提出了一些新的业务需求，遗留服务的逻辑不能完全满足。我通过阅读遗留服务代码，理清旧有逻辑关系，同时和产品经理反复讨论各种业务逻辑的合理性以及不同场景下的行为，并且与QA、前端、客户端一起及时沟通，成功地让每个开发人员对业务都有了充分的了解。

- 编写具体业务代码时，采用了akka-persistence这样一套CQRS框架。相比以前的Spring Boot内存占用降低了70%。而且akka-persistence提供的模型可以上我很轻松地在内存中线程安全地实现业务逻辑，避免了遗留系统中大量锁的存在。

- 服务使用了MySQL作为持久化存储。同一套业务数据需要同时面向C端和B端(即运营端)提供查询功能，即使数据库和表相同，两者的查询场景也有很大区别。针对这些不同场景，我使用MySQL explain命令分析SQL运行的成本，选择分别做索引进行优化或者建立一个大而全的索引进行查询性能优化。

- 迁移过程中也有其他同事参与后端相关的开发，最多时有3人。作为遗留服务迁移的负责人，我也承担了一些管理职能，将一个大任务拆解后分配给不同的同事，每天定时开会对齐进度，解答同事们关于业务或技术上的疑问。因为我最了解业务，所以我主要编写核心业务逻辑，把一些其他工作，比如和下游服务的交互分配给同事们，尽可能多地提升效率。
```
